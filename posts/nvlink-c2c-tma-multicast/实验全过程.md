## 0. 问题的由来

两个起点:

- **一条 folklore**:"NVLink 上用 TMA multicast 访问远程内存会直接让卡挂逼。"——有多人附和、看似可复现。
- **一篇论文**:DAK(*Direct-Access-Enabled GPU Memory Offloading*, arXiv:2604.26074),反其道而行,在 **NVLink-C2C** 上用 TMA + TMA multicast 把 offload 的权重/KV cache 直接读进 SMEM,号称比 prefetch baseline 最高 3×。

矛盾就在这里:如果"NVLink 上 TMA multicast 访问远端会挂",DAK 为什么能在 NVLink-C2C 上跑?最初的假说(来自社区讨论)是:

> TMA multicast 的扇出要从**本地 L2**取源;NVLink peer 内存是非一致"外部窗口",数据只 cache 在对端、绕过本地 L2,于是 multicast 找不到本地源 → mbarrier 死锁 → 挂卡。而 C2C 是一致内存、读它会落本地 L2,所以能跑。

本记录就是去**动手验证这个假说**,看它对不对。

探针 `tma_mc_probe.cu`:一个 cluster of N CTAs,CTA0 发 `cp.async.bulk.tensor.2d…multicast::cluster` 把一个 tile 扇出到 cluster 内各 CTA 的 SMEM,每个 CTA 在自己的 mbarrier 上等待;只改 **TMA 源地址的内存类型**做对照。

## 2. 实验一:C2C 上的 TMA multicast 数据通路(GH200)

**目的**:验证假说的前半句 ——"C2C 内存是挂在本地 L2 后面的一致内存,读它会落本地 L2"。

**做法**:固定一切,只改 TMA 源:
- mode 0 = 本地 HBM(`cudaMalloc`)
- mode 3 = C2C Grace 内存,强制不迁移(`cudaMallocManaged` + `cudaMemAdvise(PreferredLocation=CPU, AccessedBy=GPU)`)
- mode 4 = host-pinned mapped(`cudaHostAlloc`)

> 坑:默认 `cudaMallocManaged`+prefetch 到 CPU(早期 mode 2)在 GH200 上会 **fault-migrate 回 HBM**,读到的其实是 HBM。必须用 `PreferredLocation=CPU + AccessedBy=GPU`(mode 3)才测得到干净的 C2C 远端读。

**功能结果**:mode 0 / 3 / 4 全部 **DONE**(multicast 完成,无死锁)。

**NCU 数据通路**(32 KB tile,cluster=2,multicast):

| 指标 | mode 0 本地 HBM | mode 3 C2C(不迁移) | mode 4 host-mapped |
|---|---|---|---|
| `lts__d_sectors_fill_device`(从 HBM 填 L2) | **1204** | 207 | 199 |
| `lts__d_sectors_fill_sysmem`(从 Grace/C2C 填 L2) | **0** | **1024**(=1 tile) | **1024** |
| `dram__bytes_read` | 38.66 KB | 6.66 KB | 5.89 KB |

`fill_sysmem = 1024 sector = 正好一个 32 KB tile`:**C2C 远端读确实穿过本地 L2 填充进来**。mode 0(HBM)与 mode 3/4(C2C)的 L2 填充来源**方向相反**(device vs system),但**都落进同一个本地 L2**,所以都能作为 multicast 的本地扇出源。

> 📊 图见同目录 **`memory_chart.html`**:并排展示 mode 0(HBM→L2)与 mode 3(Grace→C2C→L2),两者都过本地 L2、再 multicast 扇出到簇内 SMEM,差别只在 L2 的填充来源。

**结论(假说前半句成立)**:C2C 内存对 TMA 引擎就是"挂在本地 L2 后面的内存",`fill_sysmem` 实测坐实。NCU 完整存档:`report_mode0_hbm.ncu-rep` / `report_mode3_c2c.ncu-rep`。

## 3. 实验二:单卡 / MIG 为何造不出"挂卡"

要验证假说的后半句(NVLink peer 会挂),需要一个**非一致 peer 外部窗口**。但 GH200 是**单卡**:

- mode 1(peer GPU)需要第二块 GPU,单卡无法构造。
- GH200 上连 host-mapped 内存(mode 4)都走 C2C 一致路径(`fill_sysmem=1024` 而非挂死),**物理上造不出非一致外部窗口**。

**尝试用 MIG 切两个实例模拟双卡**——实测结论:**模拟不出来,而且原因很有启发**。

把 GH200 切成两个 `3g.48gb` MIG 实例后:单进程能看到 2 个设备、`cudaDeviceCanAccessPeer=1`、peer access 成功;但 mode 1 仍 **DONE 不挂**:

| | mode 0(本实例 HBM) | mode 1(peer MIG 实例) |
|---|---|---|
| `lts__t_sectors_op_read` | 1792 | **1790** |
| 结果 | DONE | **DONE** |

两者 `op_read` 几乎相同 → peer 实例的数据同样落本地 L2。根因:**MIG 两个实例物理上是同一块 GPU、共享同一 L2 fabric**,peer 读仍落本地 L2。要构造真正的"独立物理 L2"外部窗口,必须两块独立物理 GPU。

(实验后已 `nvidia-smi -mig 0` 还原。)

## 4. 实验三:双卡 H100 上的反转 —— peer 也不挂

拿到 2× H100 SXM5(NVLink)后,终于能构造真正的 mode 1:**以 peer GPU 内存作 TMA multicast 的源**。这正是 folklore 说必挂的场景。

**结果出乎预料 —— 没挂卡**。探针加了"回写实际搬进 SMEM 的首字"做数据校验(`src` 全填 `0x01010101`):

| 配置 | 结果 | 每个 CTA 的 SMEM 首字 |
|---|---|---|
| mode 0 本地 HBM | 2/2 DONE | `0x01010101` |
| **mode 1 peer NVLink(cluster=2)** | **2/2 DONE** | **`0x01010101`** |
| mode 1 peer(cluster=4) | 4/4 DONE | `0x01010101` |
| mode 1 peer(单播 TMA) | DONE | `0x01010101` |

不是假完成:首字 = 正确 peer 数据,且所有 CTA(含非发起的 consumer)都完成。

**机制证据(NCU NVLink 流量)**:

| 指标 | mode 0(本地) | mode 1(peer) |
|---|---|---|
| `nvlrx__bytes`(NVLink 接收) | **0** | **40.99 KB**(≈1 tile + 协议) |
| `nvltx__bytes`(NVLink 发送) | 16 B | 25.20 KB |

完整链路:请求经 NVLink 发往 GPU1 → 数据经 NVLink 拉回本地(nvlrx=41 KB)→ TMA 写进本地 SMEM(首字校验通过)→ multicast `complete_tx` 完成 → 不挂。

**为什么假说错了**:假说把"普通 `ld` 指令读 peer 的 L2 缓存行为"错套到了 TMA 上。**TMA(`cp.async.bulk`)是主动 bulk DMA**:它从 global 地址(含 peer,经 NVLink)主动拉数据、写入本地 SMEM,`complete_tx` 由"字节实际到达"驱动,**不依赖本地 L2 是否预先持有源行**。所以源是 peer 也不会"无本地源死锁"——TMA 会把数据搬过来。

## 5. 实验四:挂卡的真正根因 —— mbarrier 协议

可"有人反映挂卡"是真实信号,不能无视。继续深挖,把挂卡稳定复现出来了,但根因**不是 NVLink**。

TMA multicast 的 `complete_tx` 会**分别发往 cluster 内每个目标 CTA 自己的 mbarrier**,所以**每个目标 CTA 都必须 `init` 并 `arrive.expect_tx` 自己的 mbarrier**。一个非常常见的错误是:**以为只有发起 CTA 需要 arm `expect_tx`**,consumer CTA 只 init 不 arm。

**2×2 对照**(探针 `-DBROKEN_BARRIER` 复现错误写法):

| mbarrier 协议 | mode 0 本地 HBM | mode 1 peer NVLink |
|---|---|---|
| **正确**(每个 CTA arm 自己的 expect_tx) | 2/2 DONE ✓ | 2/2 DONE ✓ |
| **错误**(只发起 CTA arm) | **挂死(rc=124)** | **挂死(rc=124)** |

**关键:错误协议在本地 HBM 上一样挂死,正确协议在 peer 上一样不挂 —— 挂卡与 peer/NVLink 完全无关。**

**cuda-gdb 抓到的"卡住"字面现场**(attach 挂死的 BROKEN kernel):

```
[Switching focus to ... block (1,0,0), thread (0,0,0), ... sm 125 ...]
... at tma_multicast_barrier.cu:112        # mbarrier.try_wait 自旋
  BlockIdx ThreadIdx   PC          Filename                    Line
* (1,0,0)   (0,0,0)    0x...570    tma_multicast_barrier.cu     112
=> SEL R7,RZ,0x1,!P0                        # try_wait -> selp
```

- **只有 consumer CTA(block 1)还活着,PC 死死卡在 `mbarrier.try_wait`**;
- **issuer CTA(block 0)已经退出**(cuda-gdb 报 `Invalid coordinates`)。

含义:multicast 投递本身正常,`complete_tx` 到达了正确 arm 的 issuer、它完成退出;唯独没 arm `expect_tx` 的 consumer CTA 记账永远闭合不了,死在 try_wait。这就是 folklore 里"卡住"的字面现场,而它证明的是**协议 bug,不是 NVLink**。

**为什么 folklore 会怪 NVLink**:TMA multicast 的典型场景就是跨卡/远程 offloading,大家都是在"NVLink 远程内存"这个场景里第一次写 multicast,协议没写对就挂;而没人在本地 HBM 上用 multicast,所以"本地其实也会挂"从没被注意到。**folklore 抓对了现象,抓错了归因。**

> 🔧 最小可复现示例见同目录 `tma_multicast_demo/`(`tma_multicast_barrier.cu` + `capture_hang.sh` + `README.md`):用**本地 HBM** 就能演示正确 2/2 DONE vs 错误 consumer 挂死,并用 cuda-gdb 抓现场。

## 6. 实验五:DAK 论文代码复现

DAK 有开源代码([github.com/shouxulin/DirectAccessKernel](https://github.com/shouxulin/DirectAccessKernel))。在 GH200 上 clone、构建、运行。

**构建**(踩坑):需补 CUTLASS include、装 pybind11、并 patch CUTLASS 一处 host/device 标注才能编过 torch 扩展(`offload.runtime`)。

**核心机制复现成功**(`benchmark_linear.py`,decode GEMV,bsz=8):

| 类型 | 权重位置 | m×k | 有效带宽 | MAPE |
|---|---|---|---|---|
| vanila(cuBLAS) | HBM | 4096² | **1942 GB/s** | 0% |
| DAK direct | HBM | 4096² | 1567 GB/s | 1.17% |
| **DAK direct** | **Grace(C2C)** | 4096² | **304 GB/s** | 2.05% |
| **DAK direct** | **Grace(C2C)** | 11008×4096 | **323 GB/s** | 0.88% |

DAK 的核心 —— **权重 offload 到 Grace 后不 prefetch 回 HBM,直接经 C2C 读进 SMEM 计算** —— 复现成功:达到 ~304–323 GB/s(C2C 单向 450 GB/s 的 ~70%),结果正确。这正是论文相比 prefetch 提速的根源。

**一个源码核实**:社区指出 DAK 的 `gemv_multicast_prefill` 把 TMA multicast 注释了。核对属实——该 kernel 没有 `__cluster_dims__`、实际调用的 `loop_prefill` 里两处 multicast 调用全被注释(改用非-multicast),只有 decode 路径的 `gemv_multicast` 真正用 multicast。(benchmark 的 `--type multicast` 在本机崩在该 kernel 内部的其它指令,与 multicast 无关。)

## 7. 总结

把整条因果链走完后,最初的假说被**部分证实、部分推翻**:

| 命题 | 结论 |
|---|---|
| C2C 内存读会落本地 L2 | ✅ 证实(`fill_sysmem`=1 tile) |
| C2C 上 TMA multicast 能跑(DAK) | ✅ 证实(mode 3/4 DONE;DAK direct 304–323 GB/s) |
| **NVLink peer 上 TMA multicast 必挂** | ❌ **推翻**(双 H100 实测 mode 1 稳定 DONE,数据正确,nvlrx 证明数据经 NVLink 拉回本地) |
| 挂卡的真正根因 | **mbarrier 协议误用**(consumer CTA 没 arm `expect_tx`),与 peer/NVLink 无关;本地 HBM 用错误协议一样挂死 |

**一句话**:能不能跑 TMA multicast,取决于"数据是否被搬到了本地、每个 consumer CTA 的 mbarrier 是否正确记账",而**不是**"源是不是 NVLink 远程内存"。

- C2C(落本地 L2)、本地 HBM、甚至 NVLink peer(TMA 主动经 NVLink 搬回本地)—— 只要协议写对,**都能跑**。
- 协议写错(consumer 不 arm)—— 无论源在本地还是 peer,**都挂死**,现场就是 consumer CTA 卡在 `mbarrier.try_wait`。

folklore 描述的"挂卡"是真的,但它是一个**协议 bug**穿着"NVLink 远程内存"的外衣;真正的分界线从来不是链路类型。
